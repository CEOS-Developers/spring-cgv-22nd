동시성 문제란, 멀티 스레드 환경에서 같은 값을 변경할 때 발생하는 문제이다.

❓멀티 스레드 환경❓

→ 하나의 프로세스 내에서 여러 개의 스레드가 동시에 작업을 수행하는 기술로, Spring은 기본적으로 멀티 스레드 환경에서 동작한다.

그렇다면 왜 멀티스레드 환경에서 발생하는 문제인걸까?

싱글스레드 환경에서는, 하나의 스레드가 모든 일을 처리하기에, 동시성 문제가 발생하지 않는다.

그러나 멀티스레드환경에서는 만일 한 자원에 여러 스레드가 동시에 접근할 경우, 경쟁 조건에 의해 동시성 문제가 발생한다.

이렇게 두 개의 스레드가 물건을 구매한다고 할 때, 수량 조회 후 동시에 물건을 구매했다고 가정하자. 이때 Thread 2의 속도가 조금 늦어져 Thread 1이 먼저 구매하고 나서 구매했을 때, 둘 다 수량 감소 후 99로 저장하여 최종 결과값이 98이어야하나, 99인 문제가 발생한다.

그렇다면 이를 어떻게 해결해야할까?

1. **synchronized / JVM 내부 락**

   Java 수준에서 메서드 혹은 블록에 synchronized 키워드를 걸어서 **한 JVM 내에서** 동일 객체/모니터에 대한 동시 접근을 막는 방식
   👍  구현이 간단하며, 로컬 환경에서 빠르게 적용시킬 수 있다

   👎  인스턴스가 여러 개인 환경에서는 효과가 없다

   락 범위가 너무 크면 성능 저하, 데드락 가능성 존재

2. **데이터베이스 락 (DB Lock)**

   데이터베이스 차원에서 동시성을 제어하는 방법

    - **Pessimistic Lock (비관적 락)**
      동시성 문제가 발생한다는 가정 하에 자원에 대한 접근을 막는 방식.
    - **Optimistic Lock (낙관적 락)**
      동시성 문제가 드물 것이라는 가정 하에 락을 걸지 않고 처리하다가 동시성 문제가 발생하면 그때 처리하는 방식
3. **분산 락 (Redis만 알아봄)**
   여러 분산된 서버가 존재할 때, 여러 서버에서 공유 자원에 접근하여 발생하는 Race Condition 자체를 없앤다.

   분산 락은 공유 자원 자체에 Lock을 설정하는 비관적 락/낙관적 락과 다르게 임계 영역(critical section)에 Lock을 설정한다.

   ✅Redis는 싱글 스레드 기반 → 멀티플렉싱 기술을 사용하고, 작업을 처리하는 부분이 싱글 스레드라, 원자성을 보장한다

    - Redisson

      RedLock 알고리즘을 사용해서 Lock 획득-해제를 pub-sub 구조로 수행

    - Lettuce

      Spin Lock으로 지속적으로 Redis 서버에 요청을 보내서 Lock 여부 확인


그렇다면 이번 cgv 클론 코딩에서는 어떤 로직을 사용하면 좋을까??

→ redis 분산 락이 적합하다고 판단

첫번째 이유: 우선 **synchronized**는 인스턴스가 여러 개인 환경에선 필요없다. cgv라면 무조건 인스턴스가 여러 개일 거라고 생각하여, 제외했다

DB 락 vs redis 분산 락

: 분산 락은 여러 서버가 동일 자원에 접근할 수 있는 환경에서 싱글 스레드기반이므로, 한 번에 하나만 작업하게 제어할 수 있다. 또 DB락은 DB내에서의 동시성 제어라, 애플리케이션 로직내에서 제어하는 게 더 안전하다고 판단하여 분산 락을 선택

Lettuce vs Redisson에서 Redisson을 선택한 이유

: Lettuce같은 스핀 락을 사용하는 경우, redis에 부하를 줄 수 있으나, redisson은 pub/sub 방식을 사용하기에 부하를 줄일 수 있다 그래서 redisson을 선택

### **A. 매점 상품 구매 로직**

- 시나리오: 사용자가 상품 여러 개를 담아서 구매 → 재고 차감
- 문제: 여러 사용자가 거의 동시에 동일 상품을 구매 → 재고가 음수가 되거나 과매출 발생 가능
- 흐름:
    1. 사용자가 구매 요청을 하면 상품들에 대한 락 획득(분산 락)
    2. 결제가 끝난 후에 재고 차감
- 재고 차감 케이스: **결제를 한 후**에 재고를 차감하는 방식 채택

### **B. 영화 예매 로직 적용**

- 시나리오: 사용자가 좌석을 선택하고 결제창으로 진입 → 결제 완료되기 전이라도 좌석을 확보해버리는 방식
- 문제: 여러 사용자가 동일 스케줄/좌석을 거의 동시에 선택 → 중복 예약 가능성
- 흐름:
    1. 사용자가 예매 요청을 하면 좌석에 대한 락 획득(분산 락)
    2. 결제창 진입 완료 후 좌석 RESERVED 상태로 저장
    3. 결제 실패 또는 이탈 시에 시간 제한을 둔 후 좌석 풀기
- 재고 차감 케이스: 좌석 선택 후 결제창 들어갈 때 재고를 차감하는 방식 채택
